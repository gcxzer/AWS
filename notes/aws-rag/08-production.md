# RAG-8：生产化能力：权限隔离、监控、成本、安全

## 学习目标

本阶段目标是把 RAG 从 Demo 思维推进到生产系统思维。你不一定要马上上线完整系统，但要能说明如果进入真实企业环境，需要补哪些安全、成本、监控和运维能力。

完成后你应该能设计：

- 最小权限和多租户隔离策略。
- 日志、指标和追踪方案。
- 成本估算和成本控制点。
- 敏感信息、拒答和审计机制。

## 核心理论

### 权限隔离

企业 RAG 的核心风险之一是“用户看到不该看的文档”。权限隔离必须贯穿文档上传、索引、检索和回答。

常见做法包括：

- 文档层 metadata 标记部门、角色、租户或权限组。
- 检索时强制附加权限过滤条件。
- S3、向量库、应用层权限保持一致。
- 日志和 citation 不泄露用户无权访问的来源。

### 多租户边界

多租户系统要明确隔离边界：是按租户建不同 bucket、不同 index、不同 Knowledge Base，还是在同一索引中用 metadata filter 隔离。

隔离越强，安全边界越清楚，但成本和运维复杂度越高。共享索引成本低，但必须非常重视过滤条件和测试。

### 可观测性

RAG 生产系统至少需要观察：

- 请求量、延迟、错误率。
- 模型调用 token 和成本。
- 检索命中数量、top-k、空结果率。
- 用户反馈、拒答率、citation 缺失率。
- ingestion 成功率和文档同步延迟。

没有这些指标，系统质量只能靠主观感觉。

### 成本模型

RAG 成本通常来自：

- 文档解析和 ingestion。
- embedding 生成。
- 向量存储和查询。
- 生成模型输入/输出 token。
- 日志、监控、网络和存储。

优化成本不只是换便宜模型，也包括减少无效 chunk、控制 top-k、缓存常见问题、选择合适模型和避免重复 ingestion。

### 安全与合规

企业文档可能包含 PII、商业机密、合同、财务数据或客户信息。RAG 系统必须考虑：

- 哪些文档可以进入知识库。
- 是否需要脱敏。
- 是否允许模型看到原文。
- 日志是否记录敏感内容。
- 用户是否可以通过 prompt injection 获取越权信息。

## 关键概念

- **Tenant Isolation**：租户隔离。
- **Access Control**：访问控制。
- **PII**：个人敏感信息。
- **Audit Log**：审计日志。
- **Prompt Injection**：提示注入攻击。
- **Cost Guardrail**：成本保护。
- **Observability**：日志、指标、追踪。
- **Fallback**：失败或低置信度时的兜底策略。

## 工程取舍

- 隔离越强，成本和管理复杂度越高。
- 日志越详细，排查越容易，但隐私风险越高。
- 严格拒答更安全，但可能降低用户体验。
- 更大的模型提升质量，但也提高成本和延迟。

## 动手实验

1. 为测试文档设计权限 metadata，例如 department、role、tenant。
2. 设计检索时必须应用的权限 filter。
3. 制定日志字段清单，区分可记录和不可记录内容。
4. 估算一次问答请求的 token 成本。
5. 设计无答案、检索为空、模型错误、权限不足时的返回策略。
6. 写一份 prompt injection 风险样例和防护策略。

## 验收标准

- 有一份生产化 checklist。
- 能说明权限隔离方案和风险点。
- 能给出基础成本估算方法。
- 能列出关键监控指标。
- 能设计拒答和失败兜底策略。

## 阶段产物

- 生产化 checklist。
- 权限隔离设计说明。
- 成本估算表。
- 监控指标清单。
- 安全风险和防护清单。

## 复盘问题

- 如果用户无权访问某文档，系统如何保证不会通过回答泄露？
- 哪些日志字段可能造成敏感信息泄露？
- 如何判断一次回答应该拒答？
- 成本突然升高时，你会先查哪些指标？
