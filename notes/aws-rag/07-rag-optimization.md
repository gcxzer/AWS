# RAG-7：RAG 质量优化：Metadata、Hybrid Search、Rerank、Citation

## 学习目标

本阶段目标是建立 RAG 质量评估和优化能力。你要从“感觉回答还行”升级到“能用测试集、指标和错误分类说明系统哪里好、哪里不好、为什么变好”。

完成后你应该能做到：

- 建立一组覆盖不同问题类型的测试集。
- 区分检索错误和生成错误。
- 使用 metadata、hybrid search、rerank 改善结果。
- 检查 citation 是否真的支撑答案。

## 核心理论

### RAG 评估体系

RAG 评估至少要拆成两层：

- **检索质量**：相关证据有没有被找回来。
- **生成质量**：模型是否基于证据准确回答。

如果只看最终答案，很难判断问题来自哪一层。一个系统可能检索正确但生成错误，也可能生成表达很好但引用根本不支持答案。

### Recall 和 Precision

Recall 关注“该找到的有没有找到”。Precision 关注“找到的是否真的相关”。RAG 常见优化是在二者之间找平衡：

- 提高 top-k 可能提升 recall，但降低 precision。
- metadata filter 可能提升 precision，但错误过滤会降低 recall。
- rerank 可以在召回后重新排序，提升进入 prompt 的证据质量。

### Faithfulness

Faithfulness 指答案是否忠实于证据。一个答案流畅、合理、甚至事实上正确，也可能不是 faithful，因为它包含了证据没有支持的内容。

企业 RAG 更看重 faithful answer，而不是“看起来聪明”的回答。

### Citation 可信度

Citation 需要人工或自动规则检查。最少要确认：

- 引用 chunk 是否包含答案中的关键事实。
- 回答是否存在没有引用支持的断言。
- 多个引用之间是否互相矛盾。
- 无答案问题是否被错误回答。

## 关键概念

- **Recall@k**：前 k 个结果是否包含应有证据。
- **Precision@k**：前 k 个结果中相关内容占比。
- **Answer Relevance**：回答是否回应了用户问题。
- **Faithfulness**：回答是否被证据支持。
- **Rerank**：对候选 chunk 重新排序。
- **Hybrid Search**：向量检索和关键词检索结合。
- **Golden Set**：带标准答案和证据来源的测试集。

## 工程取舍

- 测试集建设成本高，但没有测试集就无法稳定优化。
- rerank 增加延迟和成本，但可提高证据质量。
- hybrid search 更稳，但需要处理多路召回和排序融合。
- citation 检查越严格，系统越可信，但回答可能更保守。

## 动手实验

建立至少 30 个问题的测试集，覆盖：

- 精确事实问题。
- 跨段落或跨文档问题。
- 需要引用具体来源的问题。
- 知识库没有答案的问题。
- 带误导前提的问题。
- 表格、数字、版本或日期相关问题。

对每个问题记录：

- 期望答案。
- 期望证据来源。
- Retrieve 返回的 chunk。
- 最终回答。
- citation 是否支持答案。
- 错误类型。

## 验收标准

- 有一份至少 30 条的测试问题集。
- 能把错误分成检索错误、生成错误、引用错误和拒答错误。
- 能用一次参数或策略调整证明质量变化。
- 能写出优化前后对比报告。

## 阶段产物

- RAG 质量评估表。
- 测试问题集。
- 错误分类清单。
- 优化前后对比报告。

## 复盘问题

- 一个回答看起来正确，但 citation 不支持，应该算成功吗？
- 提高 recall 和提高 precision 哪个更重要？
- 什么问题适合用 rerank 改善？
- 无答案问题为什么必须进入测试集？
